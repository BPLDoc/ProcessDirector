Lesson 1: Introduction to Business Rules

Lesson Contents In this Lesson, we will discuss how to create and use Business Rules.

Business Rules

Process Director incorporates a Business Rules engine that empowers users to rapidly implement more complex business processes. Business Rules are reusable objects that evaluate a condition, or a set of conditions, and return a value based on that evaluation. Business Rules commonly return a true or false value, but they can also return a list of users, groups, string values, a single value, or run a custom task. Because Business Rules are reusable, the same Business Rule can be used in many different applications. A Business Rule can be embedded within your Process Timeline or Form to control how it should run and behave in various conditions. For instance, you can show or hide parts of a form, or skip a Timeline activity as not needed, based on the value a Business Rule returns.

For example, in our case, we want to build a Business Rule that will help our Process Timeline run properly. We have a Timeline Activity that will enable department approval for a Change of major Request. Not all departments, however, need to approve such a request. We have a Department Approval Required check box on our form that indicates whether or not a department needs to approve the request. If this box is checked, we want the Department Approval activity to run, but if it isn't, we want to skip that activity in our Process Timeline, because it isn't needed.

In order to do that, of course, we have to create the Business Rule we want to evaluate to determine whether or not that activity will run.

Creating Business Rules

To create our Business Rule, let's navigate to the Student Exercise folder. We could create our Business Rule here, of course, but, rather than having all of our objects stored in the same folder, we want to impose some organization on our application. While all of the objects in an application should be placed in the same parent folder, it's always a good idea to organize the application to store different supporting objects in their own folders. In a future lesson, we'll cover a number of reasons for doing this, but for now, let's just accept that this organizational method will be useful to us when we have more complex requirements.

So, from the Create New... menu, let's select "Folder". This selection will open the New Folder dialog box, enabling us to name the folder "Business Rules". Once we've named the folder, we can click the OK button to save and close the dialog box, and Process Director will automatically place us inside the new Business Rules folder.

Now that we have the folder to contain our Business Rules, we can, once again, go to the Create New... menu, and this time, we will select "Business Rule". Once again, we will see the Create Business Rule dialog Box. We'll name the Business Rule "Department Approval Required", then click the OK button to save the Business Rule and open the Business Rule definition.

On the Properties tab of the Business Rule definition, there are two important properties we want to set.

First, we need to set the Group in Value Picker property to "Training". Later, when we want to use this business rule, setting this value will create a "Training" menu item in the user interface, which will make it easier to find. We'll discuss this again when we want to use this Business Rule in our Timeline, so, keep it in mind.

Next, we need to set the Form Associated with Business Rule (optional) property. Just as we associated the form with the Timeline in the last module to make choosing form fields easier, we need to do the same thing here. This business rule will evaluate a form field value from our Change of Major Request form, so we can use the Object Picker to choose our form by clicking on the Build button (), then selecting the form from the Choose Form dialog box.

Selecting the form will set the correct value in the Object Picker. At this point, we want to save our configuration by clicking the Update option on the button menu at the top right corner of the screen.

Now that we're done with the Properties tab, we can click on the Values tab to configure how we want the Business Rule to work.

The Returns property determines what value the business rule will return. As mentioned previously, there are a number of different types of value that we can return. In this case, the default property is "Result of Condition". We are going to keep this default value, which will make the Business Rule return "true" or "false", based on the result of the condition we set. This is the most common way to use a Business Rule.

All we need to do now is set the Condition we want to evaluate. For this Business Rule, we want the Business Rule to look at our request form, and check to see whether the Department Approval Required field is checked or not.

As we saw in the previous module, we can access the Condition Builder by clicking the Click to create condition ... link. In the Condition Builder, click the Add Condition button to create the condition we need to configure.

By default, the condition is configured to use a form field for the evaluation, which make our job a little easier. All we need to do is click the Build button () to open the Choose System Variable dialog box. Because we associated the form with the business rule on the Properties tab, note that the Select Form Field dropdown is already conveniently configured to enable us to choose form fields from our Change of Major Request form.

We can now choose the Department Approval Required field from the dropdown, then click the OK button to close the dialog box, and return to the Condition Builder.

Notice that the Operator dropdown has automatically changed to "Checked" as the default value to evaluate.

The Operator drodown is dynamic, which means that, depending on the value we select on the left side of the condition, the options in the Operator dropdown will change. Because the Department Approval form field is a Checkbox, the Operator dropdown gives us the options of:

  • Checked
  • Not Checked
  • Any Value

If the Form field was a different type of field, we would see different options. For instance, an Input control would work quite differently.

For the Semester Hours field, for example, we would see a large number of operators to use to compare the form field on the left side of the condition to some comparison value we would enter on the right side of the condition. If we wanted to create a Business Value that checks to see whether the number of semester hours is greater than 150, we'd use the ">" operator to create the condition "Semester Hours > 150".

For the current Business Rule, however, the "Checked" operator happens to be what we want, so we can simply click the OK button to save and close the Condition Builder. When we do, our condition is now set to be evaluated by the Business Rule.

We are done configuring our Business Rule so we can click the OK menu button to save and close our business Rule. This Business Rule is now ready for use in our Timeline or form.

Summary

In this lesson, we created a Business Rule that we can use to control the operation of our process. In the next lesson, we will take a look at how to apply this Business Rule to our application to change the way it works.
